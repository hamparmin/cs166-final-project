Model

In this assignment, I coded a model for simulating crowd egress. The purpose of this model is to test different room-layouts and exit locations for the fastest evacuation. This model can be useful for establishing fire-hazards, and appropriate precautions for rooms hosting many people. My simulation is a cellular automaton that uses a grid to model objects, exits and persons in a room.

Assumptions

  1. There are four different types of entities represented in the grid: free space, obstacles (walls, tables, etc.), exits and people.
  2. Each person is a rational agent and determines the best exit for themselves, taking into consideration obstacles and other people.
  3. Time is discretized into steps, where each step corresponds to a physical step for every person.
  4. Each person can move the same amount per step. Either to a directly or a diagonally neighbouring cell.
  5. Continuous behaviour is approximated by asynchronous, random order update of all people in the room.

Parameters

  1. Layout. The simulation takes place in a pre-specified layout. This parameter is a matrix, where each entry corresponds to either free space, exits or obstacles.
  2. Number of people. For any layout, we may specify the integer number of people we want in the simulation. The model then randomly populates the layout with the exact number of people specified.

Simulation Rules

Below is a high-level pseudocode for my model.

SIMULATION():

while people inside < people outside:

        for every person:

                path=calculate next step to nearest exit()

                move person to path[first step]

                if person.location is outside:

                        people inside -=1

                        people outside +=1

This is a very simple representation and the actual model is comprised of a hierarchical class structure and implements Dijkstra’s algorithm for pathfinding. The classes specified by my model are:

  • Cell: Class to represent each grid of our cellular automaton.
  • Contains information about entity type (free space, obstacle, exit, person).
  • Contains variables for pathfinding algorithm.
  • Grid: Class to represent the entire grid of CA.
  • Contains functions to build Cell objects for each grid-point.
  • Contains functions (neighbourhood finder) to supplement pathfinding.
  • Finder: The core of the simulation
  • Contains main pathfinder functions (Dijkstra’s algorithm).
  • Accurately models human behaviour by calculating the best path for any person given obstacles and crowd.
  • Simulation: Super-class that implements simulation outlined in the pseudocode above.
  • Contains single-step and update functions for running simulation.
  • Contains extractor functions for visualization.

My model implements a pathfinder algorithm, Dijsktra’s algorithm, with a complexity of  where is the number of cells in my cellular automaton. Pathfinding becomes the dominant computation in my model considering that at each timestep  we need to calculate a path for each person  in the model. This results in an overall complexity of:

This means that running larger simulations will be very costly. A 50x50 grid, with 100 people for 1 timestep will approximately require a scalar multiple of 625,000,000 operations. This fact sets an upper limit on the scale on which I can carry out simulations. My animations for providing a visual representation of the layouts I ran on a high resolution 50x50 grid, while for computing test statistics I “shrank” my model down to 30x30.

It must be noted that these computational costs could be minimized if Dijkstra’s algorithm was only re-run for a person if its original path was rerouted only during collisions with other persons. As other obstacles, such as walls are static there should be no need to reroute persons around these. Yet, in my observations, I found that there can occur crowds so big, that an escape must be abandoned completely for another one in order for faster egress. Whether a crowd ever gets so big is an emergent phenomenon, therefore it is really difficult to predict. As of now, I cannot think of a heuristic for this suggested optimisation to be possible.

Analysis

We are interested to investigate the total time and total number of moves it takes for everyone to exit given different number of people and different width of exits. The below histograms illustrate the distribution of these measures when simulations are repeated for 20 times. Furthermore, I will also visualize mean density during regress to analyse the emergent effects on crowding.

Figure 3. Distribution of Total Timesteps for simulations with 20,50 and 100 people

Figure 4. Distribution of Average Moves for simulations with 20,50 and 100 people

As Fig. 3 and Fig. 4 suggests that with an increased number of people it takes more time and more total steps for the room to be fully evacuated. It is interesting to note, that one of the reasons that time taken might have been longer for larger groups is because individuals had to take a significantly larger amount of steps. This must be due to the “crowd effect” that people become obstacles for other people. Fig 5. illustrates this effect, which is especially salient with 300 people.

Figure 5. Exhibition Room Overcrowding Effect

I confirmed the same effects when plotting the density of people, averaged over 10 runs (each simulation ran for the mean of timesteps from the histogram) for 20, 50 and 100 people consecutively. Fig. 6 illustrates the “emptying” of the rooms, and as we can see the higher the original population, the longer it takes for it to empty. By fitting different degree polynomials over the simulated data, I was able to establish that the total time of egress  is a function of the total number of people  at the beginning of the simulation, best described by the quartic

I’ve used this relationship to predict approximate egress times for initial conditions that I would not have had the computational power to simulate. Fig. 6 illustrates these predictions with dashed lines. This is really useful in determining, whether a room is evacuable within a certain period of time, given this layout and some number of people in it.

I was also interested to find out, whether increasing the exit widths, first by one, then by two grids significantly affects the average time of egress. As Fig. 7 shows, exit-width does not affect egress time in low-density simulations. For simulations with 20 and 50 people, there is no noticeable effect on egress time. However, when we are running our simulation with 100 people, we can see that there is an almost 20% improvement of a 3-width exit over the 1-width. This tells us that we should make exits as wide as possible, especially if we expect regular, high-attendance event in our space.

Conclusion

In conclusion, I was able to demonstrate that my model is appropriate for determining crowd behaviour in a pre-specified layout during an emergency evacuation. My analysis of the model using Monte Carlo simulation of three test statistics, showed that the best way to ensure the safety for scenarios that involve a high-density of people is by having wide, emergency exits.

This model can be used to simulate virtually any kind of room layout (I have included two additional layout animations in the Appendix). For further research, additional test statistics, path-finder algorithms, and more realistic complications (such as the spread of fire, different agent-speeds) could be simulated.
